# checkpoint_completion_target

The `checkpoint_completion_target` parameter tells the database how quickly it should finish writing dirty data (changed data stored in memory) to the disk during a "checkpoint." Instead of rushing to write all the data at once, which creates a massive traffic jam (IO Spike) on the storage, PostgreSQL uses this setting to spread the work out over time. It is measured as a fraction of the total time between checkpoints. For example, if your checkpoints are set to happen every 15 minutes and your completion target is set to `0.9`, PostgreSQL will aim to stretch the writing process across 90% of that time—meaning it will slowly and steadily write the data over 13.5 minutes.

Spreading out this checkpoint I/O (Input/Output) is incredibly important for keeping your database running smoothly. If you don't spread it out—say the target is too low—the database will try to dump a massive amount of data onto the disk as fast as possible. This causes a sudden, intense spike in disk activity that can completely overwhelm your server's hardware. During these spikes, ordinary database tasks, like simple queries or user updates, get stuck waiting in line behind the checkpoint, causing the system to lag or temporarily freeze. By spreading the logic and I/O evenly across the checkpoint window, you smooth out those dangerous spikes into a predictable, gentle stream of background work. This ensures that the disk always has plenty of breathing room left to handle regular user requests quickly and efficiently.
